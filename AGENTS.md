# 项目协作约定

本文件用于约束 Codex 在本仓库内的默认工作方式。后续新增或修改代码时，应优先遵守这里的规则。

## 代码注释总原则

- 新增或修改关键代码时，默认补充简洁、合理的注释。
- 注释优先解释「为什么这样做」「业务约束」「边界情况」「兼容逻辑」「安全风险」，不要重复描述代码本身。
- 对一眼能看懂的赋值、判断、循环，不添加机械注释。
- 涉及权限、安全、缓存、事务、异步任务、并发控制、复杂条件、数据转换、第三方接口、兼容旧数据时，应优先补充注释。
- 修改已有复杂逻辑时，如果原注释已经过期，应同步更新或删除，避免注释与代码行为不一致。

## PHP 注释规范

- PHP 代码遵循项目现有风格，并结合 PHP 8.2 与 ThinkPHP 8 的写法保持类型清晰。
- 公共类、公共方法、核心服务方法、模型关键方法、复杂工具函数，应优先使用 PHPDoc 说明用途、参数、返回值和异常。
- PHPDoc 应使用明确类型，例如 `int`、`string`、`bool`、`array`、`?string`、`Collection`、`Model`，避免无意义的 `mixed`。
- 数组结构不直观时，应在 PHPDoc 中说明关键字段，例如 `array{id:int,name:string}` 或用文字解释业务字段。
- 可能抛出业务异常、验证异常、第三方接口异常的方法，应使用 `@throws` 标注触发场景。
- 控制器方法注释应说明接口目的、关键入参来源和权限要求；不要把路由、HTTP 方法等已由框架表达的信息重复写进注释。
- 服务层注释应重点说明业务规则、事务边界、幂等要求和跨表影响。
- 模型、Repository、查询构造相关注释应说明筛选条件、关联关系、软删除、租户/权限隔离等隐含约束。
- 配置、枚举、常量注释应说明业务含义和取值影响，避免只翻译变量名。

## PHP 注释示例

推荐：

```php
/**
 * 创建用户并初始化默认角色。
 *
 * 该方法会同时写入用户表和角色关联表，需要保持事务一致性。
 *
 * @param array{name:string,mobile:string,password:string} $payload 已通过表单验证的用户数据
 * @return int 新用户 ID
 * @throws \Throwable 当用户或角色关系写入失败时回滚事务
 */
public function createUserWithDefaultRole(array $payload): int
{
    // 默认角色由后台配置决定，避免把角色 ID 固定在业务代码里。
    $roleId = (int) config('admin.default_role_id');

    // ...
}
```

不推荐：

```php
// 获取用户 ID
$userId = $user['id'];

// 如果用户 ID 大于 0
if ($userId > 0) {
    // ...
}
```

## 变更要求

- 变更代码时保持范围克制，不做与当前任务无关的重构。
- 新增注释应服务于后续维护，不为了“看起来有注释”而增加噪音。
- 当代码逻辑比注释更难理解时，优先考虑简化代码结构，再补充必要说明。
- 若发现关键代码缺少注释，但本次不适合顺手修改，应在回复中提示可后续补齐的位置。
